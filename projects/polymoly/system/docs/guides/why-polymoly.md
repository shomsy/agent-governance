# Why PolyMoly Exists

Developers often hit a DevOps wall after application code is done.

Common pain:

- gateway and TLS setup
- database topology and failover choices
- observability stack setup
- reproducible runtime and deployment behavior

Typical options are all expensive:

- learn DevOps deeply
- hire dedicated DevOps help
- improvise and accept reliability risk

PolyMoly exists to remove this friction.

Core thesis:

- developers should focus on application logic,
- infrastructure should be safe by default,
- complexity should appear only when needed,
- systems should stay explainable and deterministic.

PolyMoly gives developers production-grade infrastructure without becoming DevOps experts.
