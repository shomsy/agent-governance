## Findings

### 1) HIGH — release evidence verdict could pass with incomplete mandatory lane artifacts

Symptom:
- release evidence indexing accepted a lane as `PASS` when the artifact directory existed, even if required proof files were missing or empty.

Root Cause:
- `laneStatus` validated job result and directory presence, but did not require `summary.txt`, `checks.tsv`, and `decision.txt`.

Impact:
- incomplete release proof bundles could be reported as promotion-ready.

Evidence:
- `system/tools/poly/internal/releaseops/release.go`

Risk:
- high

Resolution:
- fixed in the same iteration; lane verdicts now fail closed unless all mandatory artifacts exist and are non-empty.

### 2) HIGH — enterprise cluster target summary reported the Argo CD control-plane namespace instead of the deployment namespace

Symptom:
- enterprise summaries showed `argocd` as the cluster namespace.

Root Cause:
- YAML parsing used the first `namespace:` scalar in the file, which comes from `metadata.namespace`, not `spec.destination.namespace`.

Impact:
- operators could validate or troubleshoot the wrong namespace for promoted workloads.

Evidence:
- `system/tools/poly/internal/enterpriseops/enterprise.go`
- `features/gitops/argocd/application-stage.yaml`
- `features/gitops/argocd/application-prod.yaml`

Risk:
- high

Resolution:
- fixed in the same iteration; cluster targets now parse structured Argo Application YAML and report `spec.destination.namespace`.

### 3) MEDIUM — dashboard doctor summary could hide the actual failing checks

Symptom:
- the dashboard health card only rendered the first six doctor checks, regardless of severity.

Root Cause:
- doctor checks were sliced in input order instead of prioritizing non-pass and required failures.

Impact:
- a failed dashboard could visually hide the reason for failure inside the truncated list.

Evidence:
- `system/tools/poly/internal/dashboardops/dashboard.go`

Risk:
- medium

Resolution:
- fixed in the same iteration; visible checks are now ordered so required failures and non-pass results surface first.

### 4) HIGH — root-surface governance contract drift rejected an active canonical backlog source

Symptom:
- the `full` gate failed the root-surface contract because `BRAINSTORMING_REVIEW.md` was tracked at the repository root but not allowed by the enforced contract.

Root Cause:
- the enforced allowlist and the normative root-surface contract drifted away from the actual backlog/source policy that still promotes `BRAINSTORMING_REVIEW.md` as a root-level source.

Impact:
- canonical `full` verification could fail on a valid repository state, blocking execution for the wrong reason.

Evidence:
- `system/tools/poly/internal/architecture/checks.go`
- `system/docs/development/platform-architecture/ROOT_SURFACE_CONTRACT_V1.md`
- `TODO.md`

Risk:
- high

Resolution:
- fixed in the same iteration; root-surface enforcement and the normative contract now explicitly allow `BRAINSTORMING_REVIEW.md`.

## Final Decision

- Keep and Improve
