## Findings

### 1) HIGH — release evidence bundle did not require distribution artifacts or explicit rollback workflow metadata

Symptom:

- `release evidence-index` could emit a GO/NO-GO bundle without a bound `dist/poly/<version>` artifact set and without the exact rollback command/workflow required by policy.

Root Cause:

- `GenerateEvidenceIndex` validated runtime proof lanes only.
- distribution and rollback details lived elsewhere in docs and workflows but were not required inside the final release bundle.

Impact:

- a release candidate could look complete while still lacking executable operator guidance and artifact-level distribution proof.

Evidence:

- `system/tools/poly/internal/releaseops/release.go`
- `.github/workflows/release-proof.yml`
- `system/docs/development/governance/release-and-rollback-policy.md`

Risk:

- high

Resolution:

- fixed in the same iteration; release evidence now requires `RELEASE_VERSION` or exact tag resolution, validates `dist/poly/<version>` and `checksums.txt`, binds the current git SHA, and records explicit rollback workflow plus artifact anchors.
- `promoted-runtime-proof`, `ha-failure-demo`, and `stage-load-smoke` now emit the mandatory top-level `checks.tsv` / `decision.txt` artifacts required by the final bundle contract.
- `release-proof.yml` now builds and downloads the release distribution bundle before evidence aggregation.

### 2) MEDIUM — release-oriented evidence writes were not normalized behind one required-write helper

Symptom:

- release/distribution and resilience proof paths mixed direct `os.WriteFile` calls with stricter evidence helpers.

Root Cause:

- release-surface evolution introduced several artifact writers independently.

Impact:

- write failures still surfaced, but error semantics and operator signal were inconsistent across proof lanes.

Evidence:

- `system/tools/poly/internal/checkkit/report.go`
- `system/tools/poly/internal/releaseops/channels.go`
- `system/tools/poly/internal/resilienceops/runtime.go`

Risk:

- medium

Resolution:

- fixed in the same iteration; a shared required-write helper now backs check reporters, release channel generation, and release-related resilience artifacts.

### 3) MEDIUM — full help catalog drifted behind the real release command surface

Symptom:

- `poly help --all` omitted `restore-serving-proof` and `dist-channels` even though both commands were implemented and canonical.

Root Cause:

- the printed catalog was not updated when the release surface expanded.

Impact:

- operators had to inspect docs or source to discover the full release contract.

Evidence:

- `system/tools/poly/internal/cli/run.go`
- `system/tools/poly/internal/cli/ops.go`

Risk:

- medium

Resolution:

- fixed in the same iteration; the advanced help catalog now matches the implemented release command set and is regression-tested.

### 4) MEDIUM — production-ready documents still described a proof plan instead of the actual release state

Symptom:

- the main checklist still declared `Stable (Pre-Production)` and the CI plan still referenced outdated workflow naming and incomplete bundle scope.

Root Cause:

- execution closed the proof lanes faster than the status docs were updated.

Impact:

- operator-facing release truth drifted from the real artifact state, reducing trust in the decision record.

Evidence:

- `system/docs/development/production-ready-checklist.md`
- `system/docs/development/production-ready-ci-plan.md`

Risk:

- medium

Resolution:

- fixed in the same iteration; both documents now act as release records with exact artifact anchors, actual status, and the final release bundle shape.

### 5) MEDIUM — CLI version string remained on the pre-v3.5 placeholder track

Symptom:

- `poly --version` still returned `0.2.0-go-first` while the product wave and release artifacts were being treated as v3.5 release-grade output.

Root Cause:

- the binary version constant was never promoted when the v3.5 execution wave closed.

Impact:

- release semantics, operator messaging, and distribution artifacts could drift apart at the final handoff.

Evidence:

- `system/tools/poly/internal/cli/run.go`

Risk:

- medium

Resolution:

- fixed in the same iteration; the CLI now reports `v3.5.0`, aligning runtime identity with the release wave and distribution bundle path.
