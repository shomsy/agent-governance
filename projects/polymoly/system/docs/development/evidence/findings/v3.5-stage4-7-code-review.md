# V3.5 Stage 4-7 Code Review

Decision:
- Keep and Improve

Reviewed scope:
- commit `6cda617` (`feat: close v3.5 stages 4-7`)
- distribution/install, template marketplace, release workflows, dashboard, enterprise, and AI/advisory surfaces

Context gate:
- system/layer in scope: `tools/poly`, `.github/workflows`, product traceability docs
- runtime context: local development plus CI/release workflows
- compatibility constraints: AGENTS hard contract, CI profile contract, template/profile law, project-local install contract
- expectations: deterministic failure paths, auditability, and no weaker alternate CLI entrypoints

## Findings

### 1) HIGH — project-local install reported success without verifying the copied target digest

Symptom:
- `poly install` computed a source digest and wrote a manifest, but never verified that the copied project-local binary matched that digest.

Root Cause:
- `InstallProjectBinary` hashed the source binary only.

Impact:
- partial or corrupted local copies could be reported as successful installs.

Evidence:
- `system/tools/poly/internal/installops/install.go`

Risk:
- high

Resolution:
- fixed in the same iteration; copied target is now hashed and compared before success metadata is written.

### 2) HIGH — generated install channels were not fail-closed for checksum completeness and macOS checksum tooling

Symptom:
- channel generation could emit a Scoop manifest with an empty hash and the generated install script required `sha256sum`, which is not present by default on macOS.

Root Cause:
- no required-checksum validation and Linux-only checksum tooling assumption.

Impact:
- broken distribution artifacts and non-functional macOS install path.

Evidence:
- `system/tools/poly/internal/releaseops/channels.go`
- `.github/workflows/release-distribution.yml`

Risk:
- high

Resolution:
- fixed in the same iteration; required checksums are validated up front and install script supports `sha256sum` or `shasum -a 256`.

### 3) HIGH — template `min_cli` compatibility metadata was not enforced on install/import paths

Symptom:
- templates could advertise `min_cli`, but `template install` and `--from template` flows ignored it.

Root Cause:
- template descriptors were normalized for intent law only; CLI compatibility metadata never reached the boundary checks.

Impact:
- users could install/import templates with unsupported CLI versions and fail later in less diagnosable ways.

Evidence:
- `system/tools/poly/internal/cli/expansion.go`
- `system/tools/poly/internal/cli/product.go`

Risk:
- high

Resolution:
- fixed in the same iteration; CLI template entrypoints now enforce `min_cli` before applying template intent.

### 4) MEDIUM — dashboard and enterprise audit surfaces trimmed to the oldest events instead of the newest

Symptom:
- "Recent History" and limited audit views dropped the newest events first.

Root Cause:
- history is stored in ascending order and downstream consumers sliced from the start.

Impact:
- operators saw stale audit context in review/enterprise surfaces.

Evidence:
- `system/tools/poly/internal/dashboardops/dashboard.go`
- `system/tools/poly/internal/enterpriseops/enterprise.go`

Risk:
- medium

Resolution:
- fixed in the same iteration; both surfaces now keep the most recent entries.

### 5) HIGH — release distribution workflow lacked GHCR package-read permission

Symptom:
- the new distribution workflow authenticated to GHCR and ran full gates, but workflow permissions did not declare `packages: read`.

Root Cause:
- workflow split carried the content-write permission but missed the GHCR pull requirement from the CI contract.

Impact:
- release distribution could fail in CI despite valid code and artifacts.

Evidence:
- `.github/workflows/release-distribution.yml`
- `system/docs/development/governance/ci-profile-contract.md`

Risk:
- high

Resolution:
- fixed in the same iteration; workflow now declares `packages: read`.

## Final Decision

- Keep and Improve
