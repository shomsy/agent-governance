# How To Document

Version: 1.2.0
Status: Normative

## Documentation Rules

- Use simple, direct English or Serbian as allowed by the adopting repository.
- Keep documents operational, not theatrical.
- State purpose, contract, validation path, and rollback or recovery notes when relevant.
- Broken links and stale commands are documentation bugs.
- Documentation must not pretend a capability exists if it is not machine-verifiable.
- Prefer real paths, real commands, and real failure modes over abstract filler.

## Minimum Required Structure

Every important technical document should explain:

1. what it is
2. why it matters
3. the contract or rule
4. the life-cycle of the data or execution path
5. how to validate it
6. how to recover when it fails
